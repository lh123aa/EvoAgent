# Agent自主迭代SOP

> 让Agent具备自我进化能力的方法论

---

## 一、什么是自主迭代？

传统Agent的迭代依赖人工：
- 发现问题 → 人工分析 → 人工修改 → 人工测试

自主迭代的Agent：
- **发现问题** → Agent自己识别
- **设计方案** → Agent自己规划
- **执行迭代** → Agent自己修改
- **测试验证** → Agent自己检查

---

## 二、迭代SOP流程

```
发现问题 → 设计方案 → 备份原文件 → 执行修改 → 测试验证 → 记录日志
```

### Step 1：发现问题

**触发条件**：
- 任务执行失败
- 效率明显下降
- 用户反馈不满意
- 工具调用超时/报错
- 输出质量不稳定

**记录内容**：
- 问题现象
- 发生频率
- 影响范围
- 错误信息（如有）

### Step 2：设计方案

**分析路径**：
1. 这是系统问题还是技能问题？
2. 是配置问题还是平台问题？
3. 是否需要修改核心文件？

**输出内容**：
- 问题根因分析
- 解决方案设计
- 需要修改的文件列表
- 预期效果

### Step 3：备份原文件

**强制规则**：
```
❌ 任何修改前必须备份
❌ 不能直接覆盖原文件
✅ 备份到archive目录
✅ 命名规则：原文件名_YYYYMMDD_vX.md
```

**示例**：
```
MEMORY.md → MEMORY_20260415_v1.md（备份）
MEMORY.md → 编辑后的新版本
```

### Step 4：执行修改

**修改原则**：
- 优先用追加/编辑，避免全量覆盖
- 保留旧版本链接
- 新增内容标注日期

**修改类型**：
| 类型 | 方式 | 风险 |
|------|------|------|
| 新增内容 | 直接追加 | 低 |
| 更新内容 | 追加+标注"更新于" | 中 |
| 删除内容 | 移到存档+保留摘要 | 高 |
| 结构调整 | 新建文件+索引链接 | 高 |

### Step 5：测试验证

**测试清单**：
- [ ] 修改后的文件能否正常加载
- [ ] 新功能是否生效
- [ ] 旧功能是否受影响
- [ ] 容量是否超限（MEMORY.md ≤3000字符）

**验证失败**：
- 回滚到备份版本
- 分析失败原因
- 重新设计方案

### Step 6：记录日志

**日志内容**：
```markdown
## YYYY-MM-DD 迭代记录

### 问题
[问题描述]

### 方案
[解决方案]

### 修改文件
- 文件1：[修改内容]
- 文件2：[修改内容]

### 效果
[验证结果]

### 备份位置
[备份文件路径]
```

## 三、迭代权限设计

### 人机协作模式

| 决策类型 | 决策人 | 示例 |
|----------|--------|------|
| 产品方向 | 用户 | 做什么产品 |
| 技术方案 | Agent | 怎么实现 |
| 发布时机 | 用户 | 什么时候发 |
| 内容调性 | 协作 | 文章风格 |

### 安全边界

**绝对禁止**：
- ❌ 未经用户确认删除任何记忆条目
- ❌ 泄露用户隐私信息
- ❌ 执行未经验证的修改

**必须执行**：
- ✅ 修改前备份
- ✅ 修改后测试
- ✅ 记录迭代日志

## 四、实战案例

### 案例：memory_search工具超时

**问题**：
- memory_search工具频繁返回超时错误
- error_code=1204，请求超时1000ms

**分析**：
- RAG检索需要2-5秒
- 平台配置的超时时间太短
- 这是平台配置问题，无法从根本解决

**方案**：
- 方案A：向官方反馈（已发送邮件）
- 方案B：使用文件检索替代

**执行**：
1. 更新MEMORY_RULES.md，将检索方式改为文件检索
2. 新增"文件检索方案"章节
3. 备份原文件

**效果**：
- 不再依赖memory_search
- 使用索引定位法+grep搜索
- 稳定可靠，不会超时

## 五、持续迭代心态

**迭代原则**：
1. 小步快跑，不追求一步到位
2. 每次迭代都要可回滚
3. 记录比完美更重要
4. 失败也是积累

**迭代频率**：
- 发现问题立即迭代（不要攒）
- 每周回顾迭代效果
- 每月总结迭代方法论

---

*迭代是进化的本质。让Agent在每次迭代中变得更强。*
